What is it?

The Cup-O MUD client is a fully functional client for Multi-User Virtual Environments (MUVEs, aka MUDs, MU*s, MOOs, etc), written in the Java programming language. The Cup-O MUD applet allows embedding an active, real-time link to a MUVE system right into a web page or web system, which is immediately available to any user running a Java-capable WWW browser. Local-install and standalone versions of the Cup-O MUD client are also in the works.

For more information, please see the What is Cup-O MUD section of the Cup-O MUD documentation.

Downloading

Who needs to download and install it?

PLEASE NOTE: Unlike most peices of software, the Cup-O MUD client is not downloaded and installed on the user's machine. Instead, the Cup-O MUD java applet needs to be installed on the server which serves the web resources users will use to connect to the MUVE in question. This means that the Cup-O MUD client must be installed by the administrators of a particular MUVE system in order for the users of that system to be able to take advantage of it.

Please Also Note: Due to security restrictions inherent in the Java implementations on all current Java-capable web browsers I am aware of, Java applets (and thus the Cup-O MUD applet) are only allowed to open network connections to the same host they were downloaded from. Since the Cup-O MUD applet needs to open a network connection to the MUVE in question, this means that in order to function the applet must be installed on a HTTP/FTP/etc. server running on the same physical host as the MUVE it is to connect to.

About Java source code

As of version 1.0, source code for the Cup-O MUD client is also available. It is important to stress, however, that Java is designed to be a platform-independent executable format. What this means is you do not need to recompile the Java source code for an applet for each type of system you want to use it on. The compiled version of Cup-O MUD should automatically work under all platforms (unix, Win95, macintosh, Netscape, Internet Explorer, etc) with no modification.

How to get it

There are now two zip archives available for those who want to install the Cup-O MUD client on their web site:

  • CupOmud_1.3_exec.zip contains the executable Java class files needed to install Cup-O MUD on a web site, along with a copy of the Cup-O MUD documentation and an example HTML file showing a simple implementation.
  • CupOmud_1.3_src.zip contains the source code for the current Cup-O MUD release, along with some basic instructions on how to compile and use it. The source code is not needed unless you wish to make changes to the Java code of the client itself.

PLEASE NOTE: These ZIP files contain subdirectories and must be unzipped with their directory structure intact to work properly. Specifically, when installing Cup-O MUD on a web site, Applet.class MUST be located in a mudclient directory under the directory which contains CupOmud.class. You have been warned.

Caveats

Being in good part on the cutting edge of Java development, the Cup-O MUD client has uncovered its share of bugs in various (still evolving) systems. Some of them are actually bugs in Cup-O MUD itself, which I attempt to correct fairly quickly whenever I find out about them, but some also turn out to be bugs in web browsers or the underlying Java implementation, which I have little control over. Following is a list of those which have come to my attention thus far:

Mac Netscape

Well, what with the release of Mac Netscape 3.0, the Java implementation seems to be a little better than it used to be, though it still has some problems. The good news is that Cup-O MUD does seem to work... most of the time.

Unfortunately, I don't have access to a mac myself, and several of the people who have reported problems to me only seem to be able to reproduce them intermittently. However, here's a list of the currently known problems running Cup-O MUD under Mac Netscape:

  • The applet will come up and display properly, but will report "Connection Failed!". Netscape's Java Console will report a "java.net.SocketException: unknown error". This is one of the same problems which people experienced with Netscape 2.0.. I thought this had been fixed with 3.0, but at least one person has reported they still encounter it.
  • The applet will come up and display properly, will connect to the MUVE, and will allow sending commands to the MUVE and respond to MUVE output such as display-URL requests, but no output shows up in the output window.

Both of these symptoms are apparently due to flawed Java implementations in various versions of Mac Netscape. Anyone having problems with this is encouraged to complain to netscape about it.

Internet Explorer

I have had varying reports of problems with Internet Explorer (and most of these haven't been much more helpful than "it doesn't work"). I have tried it a couple of times and not had any problems with it myself, which leaves me in much the same situation as with mac Netscape above.. unless someone out there who is having problems can help me out, there's not a lot I can do to track down the problems...

Miscellaneous

The following problems have been reported to me. I don't have the resources to verify most of them. If anyone can provide more info or (even better) some info on how to get around these problems, please let me know. The platforms listed for each of the problems are the OS and application/java-implementation under which the problem was experienced, as reported to me. This doesn't necessarily mean these are the only platforms which have the given problem, and if people know of other platforms which experience the same behavior as described, please let me know and I'll add them to the list.

  • [Linux/Netscape] When non-english keyboard mapping facilities are in use, Java applets appear to respond differently to the keyboard mappings than the Netscape application does, causing keys to be mapped to non-english codes for Cup-O MUD input when they aren't for normal Netscape input (at least I think this is the problem that was reported to me).
